PRODUCT INTRODUCTION

Event Gadget零侵入自动埋点 + 智能分析工具

PART 01 · 起因

为什么想做一个自动埋点工具?

为支撑「双降」工作,业务侧已在子午线多个站点布设了埋点。但做核心分析时, 遇到了一系列系统性的取数障碍 — 不是某个站点的小问题,而是埋点链路本身的局限。

「双降」当前埋点现状
业务目标 采购 SKU 降价、采购降本
埋点上报
位置		 子午线 · 京me 站点 / 子午线 · 一站式工作台 / 子午线 · 采购价管理 — 分散在 3 个站点,各自规范不同
业务场景
采销触达
采购价巡检结果查看
多维比价 · 一键议价
供应商触达

四个核心痛点

PAIN 01
埋点规范不统一
各页面上报站点不同、字段约定不同,业务组件每接入一处都要单独埋点,人力成本高且容易遗漏。
PAIN 02
数据颗粒度不够
京me 站点不支持获取 ERP 明细,只能拿到页面访问总数 —「谁看了」「看了多久」「点了什么」全是黑盒。
PAIN 03
想分析的路径采不到
关键链路(推送 → 查看 → 调价 → 提交)在现有埋点上有缺失,要区分来源只能在本/上级页面额外打标识,改造成本高。
PAIN 04
分析依赖人工取数
一次分析 = 一次专项需求 + 等数 + 人工对齐口径,周期长、不可复用,业务决策跟不上节奏。
举例:当前埋点回答不了这些问题
  1. 京me 推送链路:每一条巡检结果推送给了谁(目前仅知 SKU 归属人)?推送成功了吗?成功后被查看了吗?查看后是否点击/跳转到采销工作台?
  2. SKU 查看:除了"是否查看页面"之外,在 SKU 详情上停留了多久?看了哪些字段?
  3. 点击去调价:除了"是否点击按钮"之外,点完之后用户去了哪?是否真的进入调价流程?
  4. 区分提交来源:点击提交时,如何分辨这单是巡检专员提交的,还是用户从其他入口进来提交的?
PART 02 · 工具价值

这个工具适合谁用 · 能做什么

主要面向用 Claude Code / Cursor / Lovable 等 AI 工具快速产出 HTML 原型来验证业务想法的个人开发者 / 小团队。 以前要验证一个新产品 idea 有没有人用、用户行为是不是符合预期,要么靠主观感受、要么得搭后端 + 接埋点 SDK + 配后台 — 几天才能搞定一次。 这个工具让你不在公司部署、不依赖团队埋点规范,加一行 script 就能在新做的 HTML 上拿到完整行为数据。

一行 <script> 接入 — 不用搭后端、不用接公司埋点平台、不用写一行采集代码。 用 AI 出的新 HTML 原型,5 分钟就能跑起来看真实用户怎么用、卡在哪 — 决定要不要继续投入。

典型使用场景

💡
用 AI 出了个 idea 想验证
Claude Code 帮你 5 分钟产出一个新功能的 HTML 原型,丢给 5-10 个真实用户试用, 用 Event Gadget 看他们真的点了什么、卡在哪 — 一周内决定值不值得做成正式产品。
🎯
A / B 两版方案不知道选哪个
两版 HTML 各接一行 script,设不同 app-id,各发一半人。看哪个版本完成率高、卡点少 — 用数据说话,不再凭审美吵架。
🧪
小工具/活动页快速看效果
报名页 / 调研页 / 活动落地页 — 这些一次性页面没必要走公司埋点流程, 但又想知道用户的真实行为(哪个表单字段被弃填、CTA 被点了多少次)。一行 script 就够。
🔍
现有页面想补一些采不到的数据
现有埋点只采到"是否点击了某按钮",但你想知道"点完之后用户去了哪""停留多久""鼠标在哪个区域徘徊" — 不动现有埋点,叠加 Event Gadget 即可补全。

核心能力

SDK
一行接入,零侵入
任意 HTML 页面加一行 <script>,无需改业务代码、无需感知埋点字段、无需打包构建。
AUTO
5 类事件自动采集
页面浏览、点击、曝光、表单交互、停留时长 — 默认全采,无需配置,无需手工打点。
TAB
同 URL 多 Tab 自动识别
SPA 页面用 display:none 切 Tab,SDK 自动识别为不同视图,看板自然分开,无需手工区分。
PATH
会话内路径自动还原
同一 session 内"从哪来 → 经过哪 → 到哪去"自动串联;按"经过 X 页"圈选用户,无需打 source 标识。
DASH
业务化分析看板
4 个页面回答 4 个业务问题:用了什么 / 卡在哪了 / 怎么走的 / 谁在用,顶部一句话白话结论。
SAFE
隐私自动过滤
密码 / 信用卡 / 敏感字段事件 SDK 端直接丢弃,不进入上报链路,符合最低数据收集原则。
实事求是 · 工具暂时做不到什么
  • 不是公司级正式埋点系统的替代,定位是"个人/小团队验证 idea 的轻量工具"。涉及合规、跨部门口径对齐、长期沉淀的数据需求,还是要走公司埋点平台
  • 采集的是浏览器内的真实用户行为,不采后端事件(订单流水、推送是否发送成功、ERP 明细等)
  • 没有内置 SSO / 用户体系,看板默认开放访问 — 适合内部验证,生产分享前需自行加访问控制
  • 数据存在 SQLite 单机文件,适合低至中等流量的原型验证(每天几千到几万事件无压力);要做正式产品级流量需迁移到 Postgres / 云数据库
PART 03 · 工具介绍 & 使用方法

工具长什么样 · 怎么用

工具分两部分:浏览器端 SDK(负责采集)+ 智能分析看板(负责呈现)。 以下按"接入 → 采集 → 看板分析 → 应用到双降场景"的顺序展示。

3.1 接入方式

在你的新 HTML 原型(无论是 Claude Code 出的、还是手写的、还是 Lovable 之类工具生成的)末尾,加这一行:

<!-- 接入 Event Gadget --> <script   src="https://<your-eg-host>/sg.js"   data-app-id="prototype-v1"   data-user-id="<用户标识-可选>"> </script>
  • data-app-id — 给你这个原型/页面起个名字(如 prototype-v1、landing-test-a),看板按此分流。A/B 两版用不同 app-id 即可对比
  • data-user-id — 可选。如果你的原型有登录系统,传用户标识可关联到真实人;不传则按浏览器 cookie 区分匿名访客
  • 无需 npm install、无需打包构建,纯 script 标签,即时生效

3.2 SDK 自动采集的 5 类事件

接入后,SDK 在用户浏览器内自动采集以下事件并 5 秒内批量上报:

PAGEVIEW
页面浏览
页面打开 / SPA 路由切换 / Tab 视图切换
CLICK
点击
任意元素被点击 + 元素文本自动识别
IMPRESSION
曝光
元素进入视口 + 可见比例 + 可见时长
FORM
表单交互
字段聚焦/改变/提交(密码字段自动过滤)
DWELL
真实停留
累计活跃时长(扣除 tab 切走的时间)

3.4 智能分析看板 — 4 个业务问题各一页

看板不堆图表,而是直接回答 4 个业务问题。每页顶部一句话白话结论 + 4 个 KPI + 主图 + 详情可下钻。

dashboard / 用了什么

↑「用了什么」— 顶部一句话告诉你"用户最常去 X(占 N%);Y 个功能是低使用功能",下方是按访问量排序的功能榜单

dashboard / 卡在哪了

↑「卡在哪了」— 自动识别死按钮(点了无后续)、高跳出页、表单失败,按"影响人数 × 比例"排序,定位最严重卡点

dashboard / 怎么走的

↑「怎么走的」— Sankey 图自动呈现"从哪进来 → 中间经过 → 在哪退出",任何一段路径可下钻到具体会话回放

dashboard / 谁在用

↑「谁在用」— 新人/回访/未知三段访客分布、回头率、高频访客 Top 10,可点任意访客看 ta 的完整使用链路

3.6 使用步骤

01

HTML 加一行 script

在你的 HTML 原型(AI 生成的、手写的、Lovable 等工具生成的)末尾插入 SDK,起个 app-id 即可,无需改其它代码

02

发给真实用户试用

把页面链接发给 5-10 个目标用户(同事 / 测试用户 / 真实客户),让 ta 们按日常方式操作,SDK 后台静默采集

03

看板自助查看结论

打开看板 → 选你刚才设的 app-id + 时间窗 → 5 秒内即可看到真实用户行为数据,决定要不要继续投入
